Roteiro sugerido para o video de entrega (maximo 7 minutos)

1. Apresentacao rapida
Falar que o grupo implementou um protocolo SRTP sobre UDP com Stop-and-Wait, ACK, timeout, retransmissao, controle de duplicados e rollback de arquivo incompleto.

2. Respostas teoricas curtas
Explicar em poucas frases:
- pacote UDP seguro: payload conservador de 1024 bytes para evitar fragmentacao;
- Stop-and-Wait: envia um pacote, espera ACK, retransmite se der timeout;
- Network Byte Order: Big-Endian;
- base64 x struct.pack: base64 para transportar binario em texto; struct.pack para cabecalho binario compacto.

3. Mostrar modularizacao do codigo
No cliente:
- montar_pacote()
- receber_ack()
- enviar_com_retry()
- transferir_arquivo()

No servidor:
- enviar_ack()
- remover_parcial()
- executar_servidor()

4. Demonstracao da transferencia com perda simulada
Terminal 1:
python servidor.py --saida recebido_foto.jpg --perda-dados 0.30

Terminal 2:
python cliente.py foto.jpg --perda-ack 0.30

Mostrar no terminal do cliente as mensagens:
- Timeout aguardando ACK
- Retransmissao
- ACK recebido

Mostrar no terminal do servidor:
- Pacote descartado pela simulacao de perda
- Seq gravado
- ACK enviado
- Pacote duplicado, quando aparecer

Ao final, abrir o arquivo recebido_foto.jpg para provar que nao corrompeu.

5. Demonstracao do rollback
Terminal 1:
python servidor.py --saida recebido_incompleto.jpg --perda-dados 0.00

Terminal 2:
python cliente.py foto_grande.jpg --perda-ack 0.00

Durante a transferencia, fechar o cliente com Ctrl+C.
Mostrar o servidor aguardando e, depois de 10 segundos, exibindo a mensagem de remocao do arquivo parcial.
Mostrar a pasta para provar que recebido_incompleto.jpg foi apagado.

6. Fechamento
Dizer que a confiabilidade foi implementada na aplicacao, mesmo usando UDP, por meio de ACK, timeout, retransmissao, tratamento de duplicados e rollback.
